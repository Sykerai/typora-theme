# Typora 全部主题 Git 管理设计

## 目标

停止使用同目录 `.bak-*` 文件管理主题历史，改用整个 `themes` 目录内的 Git 仓库统一保存所有主题的可恢复版本。

## 当前状态

- 实际目录：`C:\Users\史永康\AppData\Roaming\Typora\themes`。
- 根目录尚未初始化 Git；此前误在 `phycat` 子目录初始化了一个仓库。
- `phycat` 子目录内 63 个旧备份已删除，`themes` 根层另有 16 个旧备份待迁移后删除。
- 整个目录共有 118 个非备份正式文件，约 50.64 MiB。
- 系统已有 Git 2.50.1，且全局提交身份已配置。

## 方案

- 安全移除误建在 `phycat` 子目录中的嵌套仓库元数据。
- 在整个 `themes` 目录初始化 Git 仓库，默认分支为 `main`。
- 跟踪全部正式主题文件，使所有主题可以作为一个整体恢复。
- 新增 `.gitignore`，忽略 `*.bak-*`，防止旧式备份进入版本库。
- 创建当前稳定状态的基线提交，提交成功并验证可恢复后，再删除剩余 16 个旧备份。
- 不创建远程仓库、不推送、不发布。

## 安全顺序

1. 检查 `themes` 根目录、正式文件清单和嵌套仓库状态。
2. 在工作区通过 `apply_patch` 创建 `.gitignore` 暂存文件。
3. 暂时移出嵌套仓库元数据，将 `.gitignore` 安装到 `themes` 根目录，初始化 `main` 分支并提交全部正式文件。
4. 验证提交存在、全部正式文件均受跟踪、工作树无未忽略修改。
5. 解析并逐个验证所有待删除备份都位于 `themes` 目录树内且文件名匹配 `*.bak-*`。
6. 删除旧备份。
7. 验证备份数量为 0、正式文件未变化、Git 工作树干净，并删除暂时移出的错误仓库元数据。

## 后续维护

- 后续主题修改仍通过工作区暂存副本和 `apply_patch` 完成。
- 写入实际主题后进行静态验证，再在 `themes` 根仓库中查看差异并创建一次有意义的提交。
- 不再创建 `.bak-*` 文件；需要回退时使用 Git 历史恢复。
